# 数据清洗规范

## 研发流程

【强制】按标准研发流程执行，重要节点：
1. 编写技术方案
2. 评审
3. 开发
4. 自测【数据项及分片表】
5. 测试
6. 线上小范围验证【通常1条数据】
7. 正式清洗

## 数据一致性

【强制】数据清洗时，必须严格遵守数据一致性原则：
- 如果数据只需要更新，那就只能存在更新代码
- 如果数据只需要插入，那就只能存在插入代码
- 禁止同时存在不必要的更新和插入逻辑

## 批量大小选择

【强制】数据清洗时，对于批量大小选择要多维度考虑：
- 数据一致性要求
- 清洗时长
- 对业务的影响程度
- 数据重要性

**一般设置**：1-500条

## Mapper文件隔离

【强制】数据清洗与业务需求用到的Mapper文件SQL必须隔离，并写注释，防止相互影响，避免引起线上事故。

【推荐】数据清洗与业务需求用到的Mapper文件隔离，相互隔离，清晰不易出错。

## 批次处理

【强制】数据清洗必须按批次清洗数据，在清洗过程中必须按批次查询需要组装的数据，禁止循环遍历单条查询数据。

**禁止做法**：
```java
// 错误：循环单条查询
for (Long id : ids) {
    Item item = itemDao.getById(id); // 单条查询
    // 处理逻辑
}
```

**正确做法**：
```java
// 正确：批量查询
List<Item> items = itemDao.getByIds(ids); // 批量查询
// 按批次处理
```

**危害说明**：循环单条查询会造成存储中间件负载及QPS双高，甚至造成IOPS使用率高，影响其它业务正常执行。

## 执行原则

【强制】数据清洗执行必须遵守以下原则：

1. **申请要求**：向技术主管和技术经理申请执行
2. **执行时间**：晚上低峰期执行
3. **并发限制**：原则上单平台同时只能运行一个数据清洗任务
4. **前置检查**：执行之前技术主管必须确认存储中间件的CPU、QPS、IO使用率等参数是否正常，存储中间件能否再支持运行数据清洗任务
5. **特殊情况**：如果确实需要白天执行或单平台同时执行多个，必须跟技术经理沟通确认
