# 数据导入导出规范

## 执行原则

【强制】数据导入导出必须遵守以下原则：

1. **申请要求**：向技术主管和技术经理申请执行
2. **执行时间**：晚上低峰期执行
3. **并发限制**：原则上单平台同时只能运行一个数据导入导出任务
4. **前置检查**：执行之前技术主管必须确认存储中间件的CPU、QPS、IO使用率等参数是否正常，存储中间件能否再支持运行数据导入导出任务
5. **特殊情况**：如果确实需要白天执行或单平台同时执行多个，必须跟技术经理沟通确认

## 与数据清洗的区别

| 场景 | 导入导出 | 数据清洗 |
|------|---------|---------|
| 方向 | 数据的导入或导出 | 数据的修改或迁移 |
| 数据量 | 通常较大 | 通常较大 |
| 风险 | 数据格式错误、重复导入 | 数据修改错误、影响业务 |
| 隔离要求 | 需要隔离 | 必须隔离Mapper |

## 导出方案选择

【推荐】数据导出尽量用异步导出方案，需要实时导出才用同步方案。

## 深分页处理

### MySQL导出
【强制】针对MySQL表的数据导出，如果导出数据量大于1万，则必须处理深分页问题，同时要注意排序字段重复问题，如create_time可能会重复，需要使用唯一字段，如id等。

### ES导出
【强制】针对ElasticSearch索引的数据导出，如果是高频场景导出用SearchAfter+PIT，如用户场景，如果是一般场景导出用Scroll，如后台场景。

【强制】针对ElasticSearch索引的数据导出，如果导出数据量大于1万，则必须通过Scroll或SearchAfter+PIT来查询数据并处理深分页问题，同时要注意排序字段重复问题。

## 排序字段重复问题

解决排序字段重复的方案：
- 使用唯一字段（如id）作为排序字段
- 或者使用复合排序（create_time + id）
