# 索引设计规范

1. **【强制】** 业务上具有唯一特性的字段（包括组合字段）必须建立唯一索引。
2. **【强制】** 查询涉及的字段若无索引，必须加索引。若索引字段数超过 5 个，考虑改用 ElasticSearch 等方案。
3. **【强制】** 超过三个表禁止 `JOIN`。`JOIN` 的字段数据类型必须一致，且被关联字段必须有索引。
4. **【强制】** 在 `varchar` 字段上建索引时，必须指定索引长度，不宜全字段索引。通过 `count(distinct left(列名, 长度))/count(*)` 计算区分度，一般 20 长度可达 90% 以上。
5. **【强制】** 页面搜索严禁左模糊或全模糊（`LIKE '%xx'`），应走搜索引擎。
6. **【强制】** 索引命名规范：
   - 主键：`PRIMARY`
   - 唯一索引：`uk_字段名` 或 `uk_字段1_字段2_...`
   - 普通索引：`idx_字段名` 或 `idx_字段1_字段2_...`
7. **【强制】** 有 `ORDER BY` 时，利用索引有序性：`ORDER BY` 的字段应放在组合索引的最后，且避免 `WHERE` 中存在范围查询破坏有序性。
   - 正例：`WHERE a=? AND b=? ORDER BY c` → 索引 `(a,b,c)`
   - 反例：`WHERE a>10 ORDER BY b` → 索引 `(a,b)` 无法利用排序
8. **【推荐】** 使用覆盖索引（`Extra: Using index`）避免回表。
9. **【推荐】** 超大分页场景使用延迟关联或子查询优化：
   ```sql
   SELECT a.* FROM 表1 a, (SELECT id FROM 表1 WHERE 条件 LIMIT 100000,20) b WHERE a.id=b.id
   ```
10. **【推荐】** SQL 性能目标至少达到 range 级别，要求 ref 级别，最好 consts。
    - `consts`：单表最多一个匹配行（主键/唯一索引）
    - `ref`：普通索引
    - `range`：索引范围检索
    - 反例：`type=index`（全索引扫描）
11. **【推荐】** 组合索引时，区分度最高的放在最左边；等号条件列前置（如 WHERE a>? AND b=?，应把 b 放前面）。
12. **【参考】** 避免极端误解：不要一个查询建一个索引；索引会消耗空间但收益远大于成本；唯一索引不需要应用层"先查后插"。
13. **【推荐】** 确认并利用索引下推（MySQL 5.6+ 默认开启 optimizer_switch 中的 index_condition_pushdown）。
14. **【强制】** 禁止对无索引字段进行 DELETE 或 UPDATE 操作，原则上需通过 id 进行。
