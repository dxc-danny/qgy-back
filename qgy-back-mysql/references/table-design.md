# 建表与字段设计规范

## 字段命名与类型

1. **【强制】** 表达是与否概念的字段，必须使用 `is_xxx` 命名，数据类型为 `unsigned tinyint`（1 表示是，0 表示否）。任何非负数字段必须是 `unsigned`。
2. **【强制】** 数据库名、表名、字段名必须使用小写字母或数字；禁止数字开头，禁止两个下划线中间只出现数字。
   - 正例：`getter_admin`，`task_config`，`level3_name`
   - 反例：`GetterAdmin`，`taskConfig`，`level_3_name`
3. **【强制】** 表名不使用复数名词（如用 `user`，不用 `users`）。
4. **【强制】** 数据库、表及字段必须有简明知义的说明（COMMENT）。
5. **【强制】** 禁用保留字（如 `desc`，`range`，`match`，`delayed`），参考 MySQL 官方保留字列表。
6. **【强制】** 唯一索引名为 `uk_字段名`；普通索引名为 `idx_字段名`。
7. **【强制】** 小数类型使用 `decimal`，禁止 `float` / `double`（存在精度损失）。若范围超出 `decimal` 限制，拆成整数和小数分开存储。
8. **【强制】** 业务场景的价格、库存、金额等字段用 `unsigned bigint`，保留 2 位小数（例如乘以 100 存储）。禁止 `int`、`float`、`double`。统计分析、财务、税务等非业务场景用 `decimal(16,2)`，具体精度需技术经理确认。
9. **【强制】** `varchar` 长度设为 2 的 N 次方，最大不超过 5000。常见长度参考：编号 32，KEY 64，URL 255，抬头 128，姓名 32，手机 32，备注 512，公司名 128。超出 5000 的字段使用 `text`，并独立成表，通过主键关联。
10. **【强制】** 表必备三字段：`id`（主键，`unsigned bigint`，单表自增步长 1），`create_time`（`datetime`），`update_time`（`datetime`）。
11. **【强制】** 数据编码使用 `utf8mb4_unicode_ci`；需要区分大小写时使用 `utf8mb4_bin`。
12. **【强制】** 字段必须设置默认值，不能为 `NULL`。默认值：数值为 0，字符串为空串 `''`，JSON 为 `[]` 或 `{}`，或按业务定义。
13. **【推荐】** 表命名格式：`业务名称_表作用`，例如 `tiger_task`、`mpp_config`。
14. **【推荐】** 库名与应用名称尽量一致。
15. **【推荐】** `datetime` 字段默认值为 `1970-01-01 00:00:00`；`date` 字段默认值为 `1970-01-01`。
16. **【强制】** 字符串类型尽量使用 `varchar`，不用 `char`（修改 `char` 长度会触发 copy 并阻塞 DML）。
17. **【推荐】** 修改字段含义或追加状态时，及时更新字段注释。
18. **【推荐】** 允许适当冗余字段以提高性能，需满足：不是频繁修改的字段、不是 `varchar` 超长或 `text` 字段。例如商品类目名称可冗余。
19. **【推荐】** 单表行数超过 500 万或容量超过 2GB 才考虑分库分表。三年内达不到此级别的不需提前分表。
20. **【参考】** 选择合适的字符存储长度以节约空间、提升检索速度。例如：年龄用 `unsigned tinyint`，海龟年龄用 `smallint`，恒星年龄用 `int`，宇宙总年龄用 `bigint`。
